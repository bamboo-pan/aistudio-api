成熟合并流程（master 受保护仓库）

目标：把其他分支的新改动合入 master，保留可追踪记录，解决冲突并完成必要验证，同时避免本地 master 与 origin/master 因 GitHub squash 合并产生历史分叉。

一、前置检查
1. 确认当前工作区状态：
	git status --short --branch
2. 如果存在与本次合并无关的未提交改动，先暂停并确认；不要擅自提交或回滚用户改动。
3. Trellis 本次任务记录不要进入合并提交或 PR 分支；提交前必须检查 staged 文件列表。

二、同步最新 master
1. 刷新远端：
	git fetch --prune origin
2. 切到 master：
	git switch master
3. 快进到远端 master：
	git pull --ff-only origin master
4. 如果不能 fast-forward，先处理本地分叉；不要直接 merge/push master。

三、确认需要合并的分支
1. 查看远端分支：
	git branch -r --no-abbrev
2. 对每个非 master 分支检查是否已有新提交：
	git log --oneline --left-right --cherry-pick master...origin/<branch>
	git merge-base --is-ancestor origin/<branch> master
3. 已完全包含在 master 的分支跳过；只有包含 master 尚未拥有提交的分支才进入合并。

四、创建集成分支，避免污染本地 master
1. 从最新 origin/master 创建集成分支：
	git switch -c merge/<branch-name>-into-master origin/master
2. 合并目标分支并留下 merge commit：
	git merge --no-ff origin/<branch> -m "merge: <branch> into master"
3. 如果有多个需要合并的分支，按风险从低到高逐个 merge，每次冲突解决后先验证再继续。

五、冲突处理
1. 查看冲突：
	git status --short
	git diff --name-only --diff-filter=U
2. 逐个文件解决冲突，保留两边有效契约/行为，不要只选择一边覆盖。
3. 解决后检查冲突标记：
	rg "<<<<<<<|=======|>>>>>>>"
4. 标记已解决：
	git add <resolved-files>
5. 提交前检查：
	git diff --check --cached

六、验证策略
1. 至少运行与合并影响面相关的单元测试。
2. 如果合并影响跨层 API、账号、网关、图片生成、真实凭据路径或处理过冲突，运行完整 unit：
	python -m pytest tests/unit
3. 本项目要求最终真实环境验证时，在 WSL home 下创建临时目录，不要直接污染工作目录：
	/home/bamboo/aistudio-api-merge-test-YYYYMMDD
4. WSL 复制时排除 .git、.venv、__pycache__、.pytest_cache，然后安装并测试：
	AISTUDIO_ACCOUNTS_DIR=/home/bamboo/aistudio-api/data/accounts python -m pytest tests/unit -q
5. 涉及账号管理/轮询/统计时，额外做不打印凭据的 smoke，只输出账号数量和字段形状，例如 account_count、stats_count、mode、has_image_sizes。

七、提交与推送 PR 分支
1. 确认 staged 文件不包含 Trellis 本次任务目录：
	git diff --cached --name-only
2. 完成 merge commit：
	无冲突 merge 通常已经自动生成提交，不需要再运行 git commit。
	如果 merge 因冲突暂停，解决并 git add 后运行：git commit --no-edit
	如果需要手动补交合并结果，则使用清晰信息：git commit -m "merge: <branch> into master"
3. 推送集成分支：
	git push origin HEAD:refs/heads/merge/<branch-name>-into-master

八、通过 PR 合入 master
1. 本仓库 master 受保护，直接 git push origin master 会被 GH013 拒绝：必须走 Pull Request。
2. 本仓库不允许 GitHub merge commit，也不允许 rebase merge；使用 squash merge。
3. 创建 PR：base=master，head=merge/<branch-name>-into-master。
4. PR 描述中写清楚：合并来源分支、冲突文件、验证命令和结果、WSL 真实环境 smoke 结果。
5. 使用 squash merge 合入 master；master 上会留下一个 squash 提交，PR 和集成分支上的 merge commit 共同作为留痕。

九、合并后的本地同步
1. PR squash 合并完成后，不要把本地集成分支再 merge 回 master。
2. 回到 master 并同步远端：
	git switch master
	git fetch origin
	git pull --ff-only origin master
3. 如果之前误在本地 master 上做过 merge commit，远端又通过 squash 合并，可能出现 ahead/behind。修复步骤：
	a. 先确认内容是否一致：git diff --stat origin/master..master
	b. 建备份分支：git branch backup/master-before-sync-repair-YYYYMMDD-<shortsha> master
	c. 内容一致后再对齐：git reset --keep origin/master
4. 最终确认：
	git status --short --branch
	应显示 master 与 origin/master 同步，且没有意外未提交文件。

十、收尾记录
1. 汇总给用户：合并分支、PR 链接、master 最终提交、冲突处理、测试结果、WSL smoke 结果。
2. 如果有临时 WSL 目录，可以保留到用户确认；确认后再清理。
3. 不要为了“确认一下”重复 push 或重复合并；GitHub/API 明确成功后即停止相关操作。
